# 方案 D：职业知识图谱（Knowledge Graph）

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`）并列 · 供横向比较
> **一句话**：把职业经历建模成一张**图**——公司/角色/项目/技能/成就/证书是节点，"任职于/使用了/贡献于/印证"是边——按 JD 做**子图匹配**找出"通往岗位要求的最强证据路径"来选材。

---

## 1. 核心理念

A 的知识是散文页、B 的是孤立原子，两者对"关系"都是弱表达。
D 把**关系当一等公民**：一个技能在"哪些项目里被哪些成就印证过"、一段经历同时覆盖了 JD 的哪几条要求——这些都是图上的可计算结构。选材 = 在图上找证据最强、覆盖最广的子图。

---

## 2. 知识载体 / 数据模型

属性图（labeled property graph）：

```
节点：Company / Role / Project / Skill / Achievement / Credential / Education
边：
  Role  --AT-->        Company
  Role  --DURING-->     (时间属性)
  Achievement --IN-->   Project / Role
  Achievement --USES--> Skill
  Project --USES-->     Skill
  Skill --EVIDENCED_BY-> Achievement   # 技能被哪些成就印证（GR-1/GR-3 的图表达）
  * 每个节点带双语标签字段 + frontmatter（时间/标签/impact）
  * Achievement 节点保留 raw_source_id 溯源
```

---

## 3. Ingest（录入/编译）

LLM 做**实体与关系抽取**：从录入文本识别出节点（项目、技能、成就…）和边（这条成就用了哪些技能、属于哪个项目），双语化后**合并进图**（同名实体消歧/对齐）。矛盾=图上的约束冲突（时间区间重叠、同公司不同信息），命中标记（GR-2）。

## 4. Query（生成）

```
1. JD → 抽取需求 → 映射到图中的 Skill / 领域节点（含同义词归一）
2. 子图匹配：从这些目标 Skill 节点反向遍历 EVIDENCED_BY / USES，
   找出印证它们的 Achievement / Project / Role
3. 路径打分：一条经历覆盖的目标需求越多、印证越强、越近期 → 分越高
4. 选取覆盖 JD 需求最优的子图（受长度预算约束）→ 选材
5. LLM 把选中子图的节点改写成贴合 JD 的双语 bullet
6. 缺口 = JD 需求里在图中无 EVIDENCED_BY 的目标节点（GR/FR-4.4）
```

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **双语**：节点标签双语字段，出稿按需取。
- **长度**：以"覆盖最多需求的最小子图"在预算内裁剪。
- **GR-2 一致性**：时间/机构冲突表达为图约束，校验自然。
- **GR-1/GR-5**：Achievement 节点带 raw_source_id，溯源清晰。
- **副产品**：可视化的"职业图谱"（技能-项目-成就网络），用户自查与缺口一目了然。

## 6. 优势

- 关系/交叉印证天然可计算，跨经历综合能力强（补 B 的短板）。
- "一段经历覆盖多少 JD 需求"可量化 → 选材更聪明。
- 一致性校验是图约束，结构上成立。
- 自带可视化职业图谱副产品。

## 7. 代价与风险

- **建模与抽取最难**：实体消歧、关系抽取、图 schema 演化都重。
- 对小素材量是"杀鸡用牛刀"，早期收益低、复杂度高。
- 需要图存储/查询栈，工程门槛高。
- LLM 抽边出错会污染图结构，纠错成本高。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | D（知识图谱） |
|---|---|---|
| 关系表达 | 散文里的互链（弱、非结构） | 一等公民（可计算的边） |
| 选材 | LLM 读全文挑 | 子图匹配 + 路径打分 |
| 跨经历综合 | 靠 LLM 叙事 | 靠图结构（强） |
| 一致性 | Lint 文本检查 | 图约束 |
| 工程量 | 低 | 高 |

## 9. 适合谁 / 何时选它

- 素材丰富、关系复杂（多项目 × 多技能交叉印证）。
- 想要"职业图谱"可视化副产品。
- 接受较重的前期建模投入。

## 10. MVP 边界与工程量

- **MVP**：可先用"轻量图"（节点+少数关键边：Achievement-USES-Skill、Role-AT-Company），子图匹配用简单遍历，不上重型图数据库。
- **后续**：完整本体、消歧、路径优化、图谱可视化。
- 工程量：**高**（图建模 + 抽取 + 存储查询）。
