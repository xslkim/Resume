# M4 JD 分析与简历生成 — 详细需求

> 框架见 [`../REQUIREMENTS.md`](../REQUIREMENTS.md)。
> **模块职责**：解析 JD；读**全量 wiki**（[M3](M3-knowledge-base.md)）选材、定详略、润色；输出**结构化简历数据**（对齐 JSON Resume，含来源 id）交 [M5](M5-rendering.md) 渲染；产出缺口分析；管理草稿。

---

## 1. 职责与边界
- **做**：JD 关键词抽取；Query（全量上下文选材+润色）；长度/语言控制；结构化数据输出；缺口分析；草稿/迭代。
- **不做**：PDF 排版（M5）、真实性核查的执行（M6，但 M4 须配合输出来源 id）。LLM 调用经 [M7](M7-llm-orchestration.md)。

## 3. 输出数据契约（→ M5）
**形态（消除歧义，解决标准兼容）**：采用"**严格 JSON Resume 主数据 + 并行 sidecar 元数据**"。
- 主数据严格符合 JSON Resume（`basics/summary/work[]/projects[]/skills[]/education[]/certificates[]/awards[]/languages[]`；`highlights` 为 `string[]`），可被标准工具校验。双语：按输出语言渲染，主数据取对应语言。
- sidecar `_meta`（并行结构，不污染主数据）：按条目/highlight 的稳定路径索引，挂 `source`（来源 page/raw id）、`lang`；顶层挂 `gaps`（缺口+硬门槛）、`params`（length/language/region/template）、`positioning`（定位）。
- M5 只渲染主数据；`_meta` 供 M6 校验与前端展示。给一份最小示例 JSON 固化契约（实现时附）。

## 4. 关键流程
```
JD 文本 → 抽关键词
     → Query(CLI, CWD=用户wiki)：读全量 wiki + JD + 长度/语言
        → 选材 + 润色 → 结构化数据(JSON Resume + _source/_lang + _gaps)
     → 草稿预览 → (M6 后置校验) → 定稿 → (M5 渲染)
```

## 5. 依赖
- M3（读全量 wiki + 状态摘要）、M7（Query 的 CLI 调用）、M6（生成后校验）、M8（JD/生成记录）。

## 6. 验收标准
- ✅ 输入真实 JD，可选长度/语言，得到结构化简历数据。
- ✅ 选材润色未引入素材外事实（配合 M6 溯源核查通过）。
- ✅ 每条 bullet 带来源 id；输出含轻量缺口列表。
- ✅ 草稿可预览、可定稿。
- ✅ 产出遵守 §7 内容质量规范（无万金油 Summary、无动词黑名单词、技能不过载）。

## 7. 内容质量规范（产品护城河，须"过度规定"）
本工具要赢过"用户自己写"和"通用 ChatGPT"，唯一靠的是**产出内容质量**。以下为 LLM 生成/润色时的**可执行约束**：

### 7.1 好 bullet 的最小特征
- 有结果/影响（STAR 的 result），不止罗列职责；优先真实可量化（指标价值梯度：营收/成本 > 用户/规模 > 速度/效率；**禁用代码行数/提交数等坏指标**）。
- 允许纯定性，但须**具体**（具体行为+具体影响），不空泛。

### 7.2 禁用模式（黑名单）
- **Summary 反泛泛**：禁止"results-driven engineer with X years of experience in…"类万金油开头；须含成长主线 + 与该岗位的具体契合点。
- **动作动词黑名单/轮换**：避免滥用 Spearheaded / Orchestrated / Drove 等；同一份简历同一动词不过度重复。
- **空泛软技能**：不单列"优秀的沟通能力"，**用具体成就体现**（呼应 `evidenced_by`）。

### 7.3 技能区规则
- **数量上限 + 分组**（避免列 20 个像报菜名）；只列**能扛面试追问**的；level 保守自评（宁可下调一档，见 M3 §3.3）。

### 7.4 真实性 vs 竞争力（系统性张力，显式承认）
缺口分析/关键词覆盖/Summary 综合/量化/详略——**多个功能都在制造"造假压力"**，而兜底（M6-FR-09 人工确认）恰恰交给"最想拿到这工作、最可能放行夸大"的当事人，是薄闸门。因此内容规范须给"**诚实但仍有竞争力**"的写法：
- 定性也能写"强"（强在具体与影响，而非形容词）；
- 近似的真实数字**标注"约/估"**，不伪装精确；
- 技能按"能扛追问"分级，而非自评膨胀；
- 缺口优先"补真实素材"，无真实经历则建议不投/调整定位（见 M2-FR-09）。

### 7.5 难处理情境的内容策略
- **短任期 / 职业空窗**：可合并相近短岗、弱化（只留年份）、或在 Summary/经历内一句话说明（如进修/项目制），不刻意隐瞒也不放大。
- **跨行转岗**：突出可迁移技能与定位（见 M4-FR-05/13）。
