# M6 校验与护栏 — 详细需求

> 框架见 [`../REQUIREMENTS.md`](../REQUIREMENTS.md)。
> **模块职责**：把护栏（GR）从"prompt 里的一句话"升级为**实际核查**。提供生成后的**一次性后置校验**与各处的一致性/溯源/诱导防护，是贯穿 M2/M3/M4/M5 的横切模块。

---

## 1. 职责与边界
- **做**：定义全部 GR；提供 `verify_against_raw`（溯源核查）、一致性校验、ATS/长度校验、后置一次性校验流程、降级判定。
- **不做**：不生成内容、不修改 wiki/简历内容（只判定 + 标注，交回相应模块/用户）。

## 2. 护栏总表
| 编号 | 护栏 | 主要落点 |
|---|---|---|
| GR-1 | 不造假/Grounded：可溯源；**自动化只保证"有出处"，"是否夸大"为 LLM 软判定，定稿前以用户人工确认为最后闸门** | M4 输出 + 本模块核查 + 用户确认 |
| GR-2 | 一致性：时间不冲突、同机构一致、头衔年限自洽；冲突报警不篡改 | M3 Ingest + 本模块 |
| GR-3 | 量化优先（不强求）：优先真实可量化；无法量化求具体行为影响；允许定性 bullet | M2/M4 |
| GR-4 | 关键词覆盖：JD 词合理出现，不堆词、不失实 | M4/M5 |
| GR-5 | 可解释：每段可回溯来源 | M4 来源 id |
| GR-6 | 降级（不交付残品）：失败时出"完整但保守 + 增强建议"，不给残缺简历 | M4/M5 |
| GR-7 | wiki 与 raw 一致：MVP 靠 Ingest 纪律；漂移检测随 Lint 后续 | M3 |
| GR-8 | 诱导防护：抽取须回显确认才入库；提问澄清不诱导 | M2 |
| GR-9 | 注入防护：用户内容经 stdin、禁止拼命令行；prompt 注入检测（输出是否偏离 schema/出现越界标志） | M7 + 本模块 |
| GR-10 | 地区合规：按目标地区校验 basics 敏感字段是否越界（投北美带照片/年龄即拦截） | M4/M5 + 本模块 |
| GR-11 | 跨语言质量：英文产出须人工确认；校验术语正确性与"与原文事实一致" | 本模块 + 用户确认 |

## 3. 功能需求
| 编号 | 需求 | 优先级 |
|---|---|---|
| M6-FR-01 | **生成后一次性后置校验**：对草稿跑 溯源(GR-1) + 一致性(GR-2) + ATS(GR-4) + 长度，**不达标则标注问题点交回用户/M4**，不自动多轮回环。 | MVP（至少 GR-1 溯源） |
| M6-FR-02 | `verify_against_raw`：把每条 bullet 的 `_source` 回查 raw/wiki，确认确有出处。结构化匹配优先，LLM 兜底。 | MVP |
| M6-FR-03 | 一致性校验：时间区间不重叠冲突、同机构信息一致、头衔与年限自洽。 | MVP |
| M6-FR-04 | ATS/长度校验：单栏/真文本/标准标题等规则 + 是否超长（配合 M5）。 | MVP |
| M6-FR-05 | 降级判定（GR-6）：素材不足/LLM 失败时，给出"可输出的真实子集 + 未达标说明"。 | MVP |
| M6-FR-06 | 诱导防护（GR-8）：校验 M2 的回显确认关卡确实存在且生效。 | MVP |
| M6-FR-07 | 校验报告留存（每次校验的结论与问题点，便于审计 GR-5）。 | MVP |
| M6-FR-08 | `verify_against_raw` 输出区分 `has_source`（结构化可判定）与 `faithfulness_warning`（LLM 软判定，如 raw"参与"被写成"主导"）。 | MVP |
| M6-FR-09 | **定稿前人工确认闸门**：要求用户对最终简历显式确认（含英文产出逐条过目），作为真实性最后一道闸。 | MVP |
| M6-FR-10 | 注入防护（GR-9）：检测生成输出是否偏离 schema / 出现"忽略以上指令"类越界信号；配合 M7 的 stdin 传入。 | MVP |
| M6-FR-11 | 地区合规校验（GR-10）：按目标地区拦截越界的 basics 敏感字段。 | MVP |
| M6-FR-12 | 翻译一致性校验（GR-11）：英文产出与中文原文事实一致、行业术语正确、动词力度恰当。 | MVP |
| M6-FR-13 | 局部重生成后**自动复校**，回环有最大轮次上限（如 2）。 | 后续 |

## 4. 后置校验流程
```
草稿(结构化数据) →
  GR-1 verify_against_raw 每条 bullet 溯源
  GR-2 一致性检查
  GR-4 关键词/ATS 检查
  长度检查
→ 全过 → 放行定稿
→ 有问题 → 输出问题清单(指出哪条/哪类) → 交回用户或 M4 局部重生成
（不自动迭代回环；最多按用户触发）
```

## 5. 接口与数据
- 输入：M4 草稿结构化数据 + 对应 raw/wiki。
- 输出：校验报告（pass/fail + 问题清单）→ 前端/M4；审计记录 → M8 `audit_log`。
- 注意：GR-1 的语义核查依赖 LLM 时仍可能误判，故**结构化 `_source` 匹配为主、LLM 为辅**。

## 6. 依赖
- M3（raw/wiki 溯源源）、M7（必要时 LLM 兜底核查）、M8（审计留存）。被 M2/M4/M5 调用。

## 7. 验收标准
- ✅ 草稿定稿前至少跑 GR-1 溯源核查；无出处的内容被拦下并标注。
- ✅ 一致性冲突被检出报警。
- ✅ 失败场景给出真实子集 + 说明，不编凑。
