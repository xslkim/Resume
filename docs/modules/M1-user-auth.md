# M1 用户与账号 — 详细需求

> 框架见 [`../REQUIREMENTS.md`](../REQUIREMENTS.md)。
> **模块职责**：提供登录与数据隔离，保证每个用户的 raw/wiki 与结构化数据互不越权。第一阶段单用户，鉴权从简但隔离边界要正确。

---

## 1. 职责与边界
- **做**：登录鉴权、会话保持、用户标识（user_id）贯穿所有模块、按 user_id 隔离文件目录与 MySQL 数据。
- **不做**：注册开放、第三方登录、权限分级、组织/协作（均后续）。

## 2. 功能需求
| 编号 | 需求 | 优先级 |
|---|---|---|
| M1-FR-01 | 单用户登录（邮箱密码）；**单用户直接预置账号，初始口令走环境变量**。邀请码归多用户（后续）。 | MVP |
| M1-FR-02 | 登录后签发**会话令牌**，采用**服务端 `sessions` 表**（M8）存 `token_hash/expires_at/revoked_at`，支持吊销；所有请求携带并校验。基础 CSRF / 会话固定防护。 | MVP |
| M1-FR-03 | 每用户拥有独立目录 `users/<user_id>/`（raw+wiki）与独立 MySQL 行，按 user_id 强隔离。 | MVP |
| M1-FR-04 | 登出（**登出即令牌失效，不可复用**）。 | MVP |
| M1-FR-05 | **改密**（改密即现有会话失效）——作为安全基线。 | MVP |
| M1-FR-06 | 多用户与权限分级、邀请管理。 | 后续 |

## 3. 接口与数据
- 输入：登录凭据。输出：会话令牌 + user_id。
- 数据：MySQL `users` 表（见 [M8](M8-storage-data.md)）。密码以哈希存储（如 bcrypt）。

## 4. 依赖
- M8（users 表）。被所有模块依赖（提供 user_id 与隔离边界）。

## 5. 验收标准
- ✅ 能登录并保持会话。
- ✅ **登出/改密后会话不可复用**。
- ✅ 任意请求只能访问自身 user_id 的文件与数据，越权访问被拒。
