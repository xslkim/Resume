# 方案 C：多智能体流水线 + Critic 自我校验循环

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`）并列 · 供横向比较
> **一句话**：把生成拆成若干**专职智能体**协作，并设一个 **Critic（审稿）智能体**强制校验真实性与一致性，不通过就退回重做，循环到达标为止。

---

## 1. 核心理念

A/B 关注"知识怎么存"，C 关注"**生成过程怎么自我把关**"。
核心差异不在数据载体（C 的 KB 可复用 A 的 wiki 或 B 的原子），而在**用一组带工具、会自我批判的智能体跑一个可迭代的流水线**——把护栏从"prompt 里的一句话"升级成"由专职 Critic agent 用工具实际核查"。

---

## 2. 知识载体 / 数据模型

载体不挑（可挂在 A 的 wiki 或 B 的原子库之上）。C 额外要求 KB 暴露**工具接口**供 agent 调用：
- `search_kb(query)`：检索素材
- `verify_against_raw(claim)`：把某条声明回查原始素材（GR-1 校验）
- `check_consistency()`：时间/机构/头衔一致性（GR-2）
- `ats_lint(draft)`：ATS 可解析性与关键词覆盖检查（GR-4）
- `length_check(draft, budget)`：长度预算检查

---

## 3. Ingest（录入/编译）

由 **Ingest-Agent** 负责：读取新素材 → 抽取/翻译双语 → 写入 KB → 调 `check_consistency` 标记矛盾。与 A/B 的录入相比，差异在于它是"带工具的 agent 主动跑校验"，而非单次 prompt。

## 4. Query（生成）

编排式多智能体流水线（带迭代回环）：

```
1. JD-Analyst：解析 JD → 需求清单（技能/级别/行业/硬性/加分项/语言）
2. Strategy/Selector：依需求 search_kb 选材，定详略与结构策略
3. Writer：把选材写成贴合 JD 的双语草稿（受长度预算约束）
4. Critic（关键）：逐条核查——
     · verify_against_raw 每条声明是否可溯源（GR-1）
     · check_consistency 时间/头衔自洽（GR-2）
     · ats_lint 关键词覆盖与可解析（GR-4）
     · length_check 是否超长
   → 不通过则附修改意见退回 Writer / Selector
5. 循环 3–4 直到 Critic 通过或达最大轮数 → 输出草稿 + 审稿记录
```

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **护栏由 Critic 强制执行**：GR-1/GR-2/GR-4 变成 agent 用工具的实际核查动作，而非寄望于主模型自觉。
- **双语/长度**：Writer 直接产出双语并受 `length_check` 约束。
- **可解释**：每份草稿附带 Critic 的核查与迭代记录（GR-5 自带审计轨迹）。
- **GR-6 降级**：达最大轮数仍不达标 → 输出"已通过校验的子集 + 未达标说明"，不强行编凑。

## 6. 优势

- 生成质量上限高：靠自我批判迭代逼近"既贴合又真实"。
- 护栏自动化、可审计：Critic 用工具核查，留下迭代轨迹。
- 模块化：各 agent 可独立替换/升级（换更强的 Writer 等）。

## 7. 代价与风险

- **成本与延迟高**：多 agent、多轮回环 = 多次 LLM 调用（与 NFR-C1 节流目标冲突，需设最大轮数/预算上限）。
- 编排复杂：状态机/回环/失败处理都要工程化。
- 可能震荡：Critic 与 Writer 反复拉锯不收敛，需收敛策略与硬上限。
- 仍依赖底层 KB 质量；C 解决"过程把关"，不解决"知识怎么存"。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | C（多智能体+Critic） |
|---|---|---|
| 关注点 | 知识载体 | 生成过程与自我校验 |
| 生成 | 单次/少次 LLM 选材润色 | 多 agent 协作 + 迭代回环 |
| 护栏 | prompt 约束 + 人工抽查 | Critic agent 用工具强制核查 |
| 成本/延迟 | 低 | 高 |
| 审计性 | 弱 | 强（带迭代记录） |

## 9. 适合谁 / 何时选它

- 追求生成质量上限、能接受较高成本与延迟。
- 看重"可审计的把关过程"（如对真实性零容忍的场景）。
- 团队具备 agent 编排能力。

## 10. MVP 边界与工程量

- **MVP**：可先做"单轮 Writer + 单轮 Critic（仅 GR-1 溯源核查）+ 最多 N 轮回退"，KB 复用 A 或 B。
- **后续**：更多专职 agent、并行批判、工具扩展、预算/轮数自适应。
- 工程量：**高**（编排 + 工具 + 收敛控制）。
