# 方案 G：大师简历 + 裁剪（Master Resume + Tailoring）

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`）并列 · 供横向比较
> **一句话**：知识库就是一份不断长大的"**大师简历**"（把所有经历成就全写进去的全集长文档）；每次投递=LLM 从中**裁剪/重排/改写**出一个针对该 JD 的子集。

---

## 1. 核心理念

A 是"wiki 形"（互链页面）、B 是"原子集合"、D 是"图"——它们的知识形态都和最终产物（简历）**不同构**，存在转换损耗。
G 让知识形态与产物**同构**：你维护的就是一份"全集简历"，每次申请只是"一次精修的裁剪"。心智模型对用户**零学习成本**——人人都懂简历。

---

## 2. 知识载体 / 数据模型

```
Master Resume（单一结构化长文档）：
  按标准简历分区组织：Summary / Experience / Projects / Skills / Education / Credential
  每个条目是一段带轻量元数据的内容：
    Bullet { text{zh,en}, tags[], metrics, 可见性默认 }
  "全集"=把你做过的一切都写进对应分区，不删减
```

与 G 的精神一致：维护**一份**文档，而非一堆页面/原子/图。

## 3. Ingest（录入/编译）

用户把新经历**追加进大师简历对应分区**（结构化条目）。LLM 在此做双语化与轻量打标签。无需"编译成另一种形态"——录入即产物的超集。

## 4. Query（生成）

```
1. JD → LLM 理解岗位要求
2. 裁剪（tailoring）：从大师简历中
     · 删：与 JD 无关的条目
     · 排：相关的提前、详写；弱相关压缩或删
     · 改：措辞贴合 JD 用词（内容不出大师简历之外）
   受用户所选长度（一页/两页）约束
3. 产出针对性简历子集 → 草稿
4. 缺口 = JD 要求但大师简历里没有的内容
```

> 因为输出是输入的**子集+改写**，"不引入大师简历之外的事实"这一护栏（GR-1）相对容易守住。

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **双语**：大师简历存双语条目，裁剪时按 JD 语言取。
- **长度**：裁剪本质就是"删到目标长度"，长度可选天然契合。
- **ATS**：大师简历本就是简历结构，套 ATS 模板顺滑。
- **GR-1**：输出是全集的子集，溯源="它在大师简历里"，较易守。
- **规模注意**：大师简历长大后，整篇塞进上下文裁剪会**重蹈 A 的上下文上限问题**（见代价）。

## 6. 优势

- 心智零学习成本：用户维护的就是"一份简历"。
- 制品同构、转换损耗小：录入→产物几乎无形态转换。
- 维护直观；规模中等时可整篇入上下文，实现简单。
- 适合"我已有一份很长的简历，想智能裁剪"的存量用户。

## 7. 代价与风险

- **规模天花板**：单文档长大后仍撞上下文上限（与 A 同病，B/D 无此问题）。
- 结构化/检索弱：靠"一篇文档"难做精细打分与覆盖率分析。
- 元数据轻 → 选材较粗，缺口分析不如 B/D 精确。
- 跨经历综合一般（裁剪是删改，不是重新连点成线）。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | G（大师简历+裁剪） |
|---|---|---|
| 知识形态 | wiki 互链页面 | 单一简历形长文档 |
| 与产物关系 | 异构（需选材综合） | 同构（裁剪子集） |
| 用户心智 | 要理解 wiki 概念 | 就是"一份简历" |
| 选材 | 全量上下文 LLM 挑 | 整篇裁剪 |
| 规模上限 | 受窗口约束 | 同样受窗口约束 |

## 9. 适合谁 / 何时选它

- 想要极简心智、不愿学"知识库/wiki"概念的用户。
- 已有一份长简历、诉求是"按岗位智能裁剪"。
- 素材量中等、暂不担心上下文上限。

## 10. MVP 边界与工程量

- **MVP**：结构化大师简历录入 + 整篇入上下文 + LLM 裁剪/重排/润色 + 长度选项 + 单 ATS 模板。
- **后续**：分区按需载入以缓解规模、裁剪偏好记忆、版本对比。
- 工程量：**低-中**（接近 A，省去 wiki 编译/互链维护）。
