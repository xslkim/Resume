## 0. 参考资料

- Andrej Karpathy, *LLM Wiki — A pattern for building personal knowledge bases using LLMs* (GitHub Gist, 2026-04)。  
  核心思想：不再用 RAG（每次查询临时从原始文档检索拼装），而是让 LLM **增量构建并维护一个持久化、互相链接的 wiki**——知识"编译一次、持续更新"，随素材增长而**复利积累**。三层结构（Raw sources / Wiki / Schema）+ 三操作（Ingest / Query / Lint）。

> 本项目把该模式**具体化到简历场景**：原始经历/文章/证书作为不可变 raw sources，LLM 将其"编译"成一个**个人职业 wiki**；投递时输入 JD，LLM 读 wiki 选材并综合出针对性简历。

## 1. 项目概述

### 1.1 背景与问题
求职者面对不同岗位，需要把同一批个人经历重新组织、取舍、润色，生成侧重点不同的简历。手工完成费时，且容易：漏掉与岗位相关但被埋没的经历；措辞与岗位描述脱节、关键词不匹配而被招聘机器人（ATS）过滤；反复重写、版本混乱。

### 1.2 产品定位
一个**个人素材沉淀库（LLM Wiki）+ 工作描述驱动的选材综合 + PDF 排版输出**的服务。

一句话：**把你所有的真实资料沉淀进一个由 LLM 持续维护的个人 wiki，告诉我目标岗位，我给你一份针对性的简历。**

### 1.3 核心价值主张
- **针对性**：同一批素材，针对不同岗位组合出不同侧重与详略。
- **真实性**：所有内容基于你提供的真实素材生成，不凭空编造。
- **复利积累**：知识库不是临时检索，而是越用越丰富、越用越准（LLM Wiki 的核心红利）。
- **省力**：素材录入到成稿，自动编译、选材、润色、排版。
- **ATS 友好**：输出能被招聘系统解析、关键词达标。

## 2. 利益相关者与用户

### 2.1 目标用户
- **主要用户**：求职者本人（素材所有者）。
- **次要用户**：小范围受邀用户（亲友等），各自拥有独立素材空间。

### 2.2 使用范围定位（已确认）
**小范围分享**：少数几个用户，需要轻量账号与数据隔离，但不面向公众开放，不做规模化运维。

> 后续如转为公开产品，鉴权、合规、运维等增量需求需另起文档评估。

## 3. 核心概念与术语

为避免歧义，统一定义关键概念。**加粗**为本版（v0.2）因引入 LLM Wiki 而新增/调整的概念。

| 术语 | 定义 |
|---|---|
| **Raw sources（原始素材）** | 用户录入的所有个人资料的**不可变**副本，是整个系统的**唯一事实源（source of truth）**。LLM 只读不写。 |
| **Wiki（个人职业 wiki）** | LLM 基于 raw sources 增量"编译"并持续维护的、结构化、互相链接的页面集合。用户阅读它，LLM 写它。随素材增长而复利积累。 |
| **Schema（规则文件）** | 告诉 LLM「wiki 如何组织、有哪些约定、录入/查询/维护时遵循什么流程」的配置文件，使 LLM 成为有纪律的 wiki 维护者而非通用聊天机器人。 |
| **Wiki page（wiki 页面）** | wiki 中的一个单元。本项目主要类型：经历实体页（一段教育/工作/项目）、成就颗粒条目、技能/证书页、主题摘要页、综合页。 |
| **Frontmatter（页面元数据）** | 每个 page 附带的结构化字段（如起止时间、机构、头衔、标签），用于过滤/排序/校验，区别于正文叙述。 |
| 成就颗粒（Bullet） | 系统的最小可复用单元：一条可独立挑选的、量化的成就描述（如"用 X 技术，将 Y 指标从 A 提升到 B"）。在 wiki 模式下作为 page 内的结构化条目或独立颗粒页存在。 |
| 标签（Tag） | 附加在 page/颗粒上的分类标记（技能/行业领域/资深程度/软技能），用于检索过滤与排序。 |
| **Ingest（录入/编译）** | 用户投喂新素材后，LLM 读取、抽取关键信息、整合进现有 wiki（更新相关页面、维护交叉引用、标记矛盾）的过程。 |
| **Query（查询/生成）** | 用户输入 JD 后，LLM 读 wiki 选材并综合出简历草稿的过程；好结果可回填为 wiki 新页。 |
| **Lint（健康检查）** | 定期让 LLM 自检 wiki：矛盾、过时、孤儿页、缺失交叉引用、数据缺口。 |
| 工作描述（JD） | 用户输入的目标岗位描述文本，是生成的触发条件与选材依据。 |
| 画像（Profile） | 用户预设的某一求职方向及其默认选材倾向（如"后端工程师""技术主管"）。同一用户可有多个。 |
| 草稿（Draft）/ 定稿（Final） | 生成结果的两个阶段：草稿可反复局部重生成，定稿为可导出 PDF 的稳定版本。 |
| ATS | Applicant Tracking System，招聘管理系统。简历需可被其解析、关键词可达标。 |
| 护栏（Guardrail） | 强制约束生成不越界的规则（见 §8），最重要的是"不造假"。 |

## 4. 范围划分

### 4.1 MVP（最小可行版本）—— 必须先打通的端到端链路
1. 轻量账号（登录、数据隔离）；
2. **手动录入**原始素材（基础字段：教育/工作/项目经历 + 成就 + 技能 + 标签），**单语输入即可**，作为 raw sources；
3. **Ingest**：LLM 把 raw sources 编译进个人 wiki，并**自动翻译成双语存储**（生成经历页/颗粒/技能页 + index）；
4. JD 文本输入；
5. **Query**：把**全量 wiki 塞进上下文**，LLM 选材 + 润色，生成草稿（基于真实素材）；
6. 生成时**简历长度可选**（一页/两页等）；输出**单套 PDF 模板**，按 JD 语言出中文/英文/双语（因 wiki 已存双语）。

> MVP 成功标准：录入若干真实素材并完成编译后，输入一个真实 JD，能在数分钟内得到一份**内容真实、针对该岗位、可投递**的 PDF 简历。

### 4.2 后续阶段（按价值排序，非 MVP）
- 多套 PDF 模板（技术 / 设计 / 学术 / 商务）；
- 缺口分析（JD 要求但你素材薄弱的点）；
- 求职信（Cover Letter）生成；
- **Lint** 健康检查（矛盾/过时/孤儿页/缺口）常态化；
- 生成结果回填复用（针对某 JD 的好版本沉淀为 wiki 页，下次复用）；
- 非文字素材：图片（作品截图/证书）、文章 URL；
- 草稿局部重生成与版本历史；
- 画像管理（多求职方向默认侧重）。

> 说明：双语已在 MVP 由 Ingest 自动翻译存储（见 FR-3.2）；向量检索/index 钻取、批量导入（旧简历 PDF/DOCX 解析）、URL 抓取、OCR 等自动化录入**不在核心路径**。录入以用户手动输入为主（见 §5.2）。后续若 wiki 长大到超限，再评估压缩/分段/索引策略。

## 5. 功能需求

> 编号规则：FR = 功能需求。优先级：**MVP** / **后续**。  
> 加 ⭐ 的为本版因 LLM Wiki 引入的新增/重写条目。

### 5.1 账号与隔离
| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-1.1 | 用户可注册/登录（邮箱密码 或 邀请码，二选一即可）。 | MVP |
| FR-1.2 | 每个用户拥有**独立的 raw sources + wiki**，相互隔离，不可越权访问。 | MVP |
| FR-1.3 | 支持登出、改密等基础账户操作。 | MVP |

### 5.2 素材录入（以用户手动输入为主，只要基础字段；单语输入即可）

> **录入原则（v0.2 明确）**：以**用户手动输入**为唯一主路径；**只要求必要的基础字段**，不强迫填满所有可选项；不做 OCR/URL 抓取/批量解析等自动化录入（至少不在核心路径）。降低使用门槛，先有最小可用素材即可编译出 wiki。  
> **双语录入（v0.3 明确）**：用户用**任一种语言**（中文或英文）输入基础字段即可；**Ingest 阶段由 LLM 自动翻译成另一种语言**，以双语形式存入 wiki。用户无需手写双语，录入负担不变。

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-2.1 | 用户可手动录入经历：教育 / 工作 / 项目，**基础字段**为机构名、起止时间、角色/头衔；其余可选。**单语输入即可**。 | MVP |
| FR-2.2 | 用户可在每段经历下录入多条**成就颗粒**，每条**基础字段**为量化描述；标签可选。**单语输入即可**。 | MVP |
| FR-2.3 | 用户可维护技能、证书、奖项、语言能力等条目（基础字段即可）。 | MVP |
| FR-2.4 | 所有素材可编辑、删除（软删除，可恢复）。 | MVP |
| FR-2.5 | 素材带录入/更新时间戳。 | MVP |
| FR-2.6 | 预置标签分类（技能/行业/资深程度/软技能），用户可自定义；每条颗粒/page 可打多标签。 | MVP |
| FR-2.7 | 素材历史版本保留（git 式，不覆盖旧版）。 | 后续 |

### 5.3 ⭐ 知识库构建与维护（LLM Wiki 模式）

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-3.1 | 用户录入的原始素材作为 **raw sources 不可变保存**，是唯一事实源。 | MVP |
| FR-3.2 | **Ingest**：每录入/更新一批素材，LLM 读取并将其整合进用户的 wiki——生成/更新经历页、成就颗粒、技能页，维护交叉引用；**并将文本自动翻译为双语（中+英）一并存储**。一次 ingest 可触及多个页面。 | MVP |
| FR-3.3 | 维护一份 **index**（内容目录：列出所有页面 + 一句话摘要 + 元数据）和一份 **log**（按时间追加的录入/查询/检查记录），便于 LLM 与用户导航。 | MVP |
| FR-3.4 | wiki 维护一套 **schema**（页面类型、字段约定、ingest/query/lint 流程），作为 LLM 维护 wiki 的纪律约束。 | MVP |
| FR-3.5 | ingest 时若新素材与已有内容**矛盾**（如时间冲突、信息不一致），LLM 予以标记，不静默覆盖。 | MVP |
| FR-3.6 | **Lint**：定期/按需让 LLM 自检 wiki——找矛盾、过时声明、孤儿页（无入链）、缺失交叉引用、重要概念缺页、可补的数据缺口。 | 后续 |
| FR-3.7 | **回填**：query 产生的好结果（如针对某 JD 的精修段落）可作为新页面沉淀进 wiki，供下次复用，让知识库复利增长。 | 后续 |

### 5.4 JD（工作描述）分析
| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-4.1 | 用户可输入 JD 全文文本。 | MVP |
| FR-4.2 | 系统从 JD 抽取：技能关键词、资深级别、行业领域、硬性要求、加分项、JD 语言（中/英）。 | 后续（MVP 可仅抽取关键词） |
| FR-4.3 | 系统计算 wiki 素材对该 JD 的**匹配度评分**（覆盖率）。 | 后续 |
| FR-4.4 | 系统输出**缺口分析**：JD 要求但素材薄弱/缺失的点，提示补充或弱化。 | 后续 |

### 5.5 ⭐ 简历生成（基于 wiki 的 Query；全量塞进上下文）

> **检索边界（v0.3 明确）**：Query 采用**纯 LLM Wiki 模式——把整个 wiki（或当前用户 wiki 的全量内容）塞进 LLM 上下文**，不做向量检索、不做 index 定位+局部钻取。理由：MVP 阶段 wiki 规模有限、现代模型上下文足够；模式最简单、信息无遗漏。后续若 wiki 长大到超限，再评估压缩/分段/索引策略。

| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-5.1 | 输入 JD 后，LLM **读入全量 wiki 内容**，从中选材（基于相关性 + 标签过滤）。 | MVP |
| FR-5.2 | 根据相关性决定**详略**：强相关展开，弱相关一笔带过，**受用户选择的简历长度约束**（一页 / 两页等可选）。 | MVP |
| FR-5.3 | 对所选内容做**润色改写**，使措辞贴合 JD 用词，但**内容必须基于真实素材（raw sources / wiki）**。 | MVP |
| FR-5.4 | 用户在生成时可**选择简历长度**（如一页 / 两页 / 简洁版），系统据此控制选材量与详略。 | MVP |
| FR-5.5 | （可选）按 JD 重写求职目标陈述（Summary/Objective）。 | 后续 |
| FR-5.6 | 按 JD 优先级重排技能区。 | MVP |
| FR-5.7 | 按 JD 语言输出中文版 / 英文版 / 双语版（因 wiki 已存双语，按需选用，无需现译）。 | MVP |

### 5.6 输出
| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-6.1 | 生成结果可导出为 **PDF**。 | MVP |
| FR-6.2 | 提供至少一套 **ATS 友好**版模板：文字可解析、关键词达标、避免纯图/复杂分栏导致解析失败。 | MVP |
| FR-6.3 | 提供多套模板（技术 / 设计 / 学术 / 商务）。 | 后续 |
| FR-6.4 | 可选导出 DOCX（供手动二次编辑）。 | 后续 |
| FR-6.5 | 可选生成针对性求职信（Cover Letter）。 | 后续 |

### 5.7 草稿与迭代
| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-7.1 | 生成结果先以**草稿**呈现，可预览。 | MVP |
| FR-7.2 | 草稿可**局部重生成**（"这段再改""换条经历"）。 | 后续 |
| FR-7.3 | 草稿确认后转为**定稿**，可导出 PDF。 | MVP |
| FR-7.4 | 同一 JD 的多次生成保留**版本历史**，可对比/回滚。 | 后续 |

### 5.8 画像管理
| 编号 | 需求 | 优先级 |
|---|---|---|
| FR-8.1 | 用户可创建多个**求职画像**（各带默认选材倾向与模板偏好）。 | 后续 |
| FR-8.2 | 生成时可套用某画像，快速获得该方向默认侧重。 | 后续 |

## 6. 非功能需求

> 编号规则：NFR。

### 6.1 易用性
| 编号 | 需求 |
|---|---|
| NFR-U1 | 从素材录入到首次出简历，关键路径步骤尽量少；录入只要求基础字段。 |
| NFR-U2 | JD 输入到草稿生成，响应在可接受范围内（建议分钟级以内，具体值待定）。 |
| NFR-U3 | 录入界面鼓励量化输入（如提示"有没有数字/成果"），但不强制。 |

### 6.2 可靠性
| 编号 | 需求 |
|---|---|
| NFR-R1 | raw sources 与 wiki 均持久化，录入/编译失败不丢数据。 |
| NFR-R2 | 生成或 LLM 调用失败时，不损坏已有素材、wiki 与草稿。 |

### 6.3 安全与隐私（v0.2 已放宽）
| 编号 | 需求 |
|---|---|
| NFR-P1 | **已确认：素材可全量发送给 LLM，直接调用线上模型**，无需本地/私有模型降级。 |
| NFR-P2 | 用户数据相互隔离，杜绝越权访问。 |
| NFR-P3 | 传输加密等基础保护。 |
| NFR-P4 | （可选）向用户告知数据会经线上模型处理；记录用了哪个模型，便于追溯。 |

### 6.4 成本
| 编号 | 需求 |
|---|---|
| NFR-C1 | LLM Wiki 模式天然省成本（编译一次、多次复用，而非每次查询重算）；仍需对重试/重生成做节流。 |
| NFR-C2 | 单次生成/编译成本可追踪（是否对用户可见，待定）。 |

### 6.5 可维护与可演进
| 编号 | 需求 |
|---|---|
| NFR-M1 | wiki 为 markdown + git 式，天然具备版本历史与可演进性。 |
| NFR-M2 | 需求分 MVP / 后续，架构应允许后续功能渐进加入，不推倒重来。 |

## 7. 约束与假设

### 7.1 约束
- **真实性红线**：系统只能组合、改写、取舍 raw sources 中的**真实素材**，绝不可凭空生成不存在的经历、头衔、技能、数据。wiki 是 raw sources 的编译产物，不得引入 raw sources 之外的事实。
- **raw sources 不可变**：是唯一事实源，LLM 只读不写。
- **数据归属**：所有素材归属于用户本人。
- 使用范围为小范围分享，不做公网规模化服务。

### 7.2 假设
- 用户愿意录入真实资料（一次性投入，长期复用）。
- 用户能判断生成结果是否如实反映自身经历，并对最终投递内容负责。
- JD 文本基本可读、非纯图片。
- 外部 LLM 服务可用且质量稳定（已确认走线上模型）。

---

## 8. 护栏（生成质量与真实性约束）⭐

这是本产品区别于"随便用 AI 生成"的核心，必须前置定义并贯穿全程：

| 编号 | 护栏 |
|---|---|
| GR-1 | **不造假 / Grounded**：所有生成内容必须可追溯到 raw sources 或 wiki 中的真实素材。无法溯源的内容不得出现在简历中。 |
| GR-2 | **一致性校验**：时间不重叠/不冲突；同一机构信息一致；头衔与年限自洽。冲突时报警（ingest 阶段标记、生成阶段拒绝），而非自行篡改。 |
| GR-3 | **量化优先**：鼓励保留/补强数字成果；纯定性描述提示用户量化。 |
| GR-4 | **关键词覆盖**：JD 关键技能词应在简历合理出现（服务 ATS），但不得为堆词塞入无关或失实内容。 |
| GR-5 | **可解释**：简历中每段内容应能回溯到来源素材/页面（便于核对）。 |
| GR-6 | **降级**：素材不足或 LLM 失败时，宁可输出不完整但真实的版本，不可"编凑"。 |
| GR-7 | **wiki 与 raw 一致**：wiki 是 raw sources 的忠实编译；Lint 负责发现并修复 wiki 内部的矛盾与漂移，防止"编译失真"成为错误事实来源。 |

---

## 9. 验收标准（MVP）

1. ✅ 用户能注册登录，raw sources 与 wiki 空间相互隔离。
2. ✅ 用户能**手动录入**基础素材：教育/工作/项目经历 + 成就颗粒 + 技能 + 标签（只要求基础字段，**单语输入即可**）。
3. ✅ 录入后系统能 **ingest**：LLM 将素材编译进个人 wiki，并**自动翻译为双语**存储（生成经历页/颗粒/技能页 + index）。
4. ✅ 用户能输入一段真实 JD 文本。
5. ✅ 用户能**选择简历长度**（一页/两页等）。
6. ✅ 系统能将**全量 wiki 塞进上下文**进行选材，做贴合 JD 的**润色**，且**未引入 raw sources 之外的事实**（人工抽查通过）。
7. ✅ 能按 JD 语言导出**中文/英文/双语 PDF**（因 wiki 已存双语），排版整洁、ATS 可解析。
8. ✅ 生成内容中每条经历都能在 raw sources / wiki 中找到来源。

## 附录 A：典型使用流程（MVP）

```
1. 注册/登录
2. 手动录入基础素材（单语即可：教育/工作/项目 + 成就 + 技能 + 标签）→ 存为 raw sources
3. Ingest：LLM 将素材编译进个人 wiki，并自动翻译为双语存储（经历页/颗粒/技能页 + index）
4. 输入目标岗位 JD 文本
5. 选择简历长度（一页/两页等）与输出语言
6. Query：全量 wiki 塞进上下文 → LLM 选材 → 润色 → 生成草稿
7. 预览草稿（核对真实性）
8. 确认定稿 → 导出 ATS 友好 PDF（中文/英文/双语）
```

## 附录 B：完整使用流程（含后续功能）

```
1. 注册/登录
2. 手动录入素材 → raw sources
3. Ingest → LLM 编译/更新 wiki（标记矛盾）→ 维护 index / log
4. 定期 Lint 健康检查（矛盾/过时/孤儿页/缺口）
5. 维护多个求职画像
6. 输入 JD → 分析（关键词/匹配度/缺口）
7. 套用画像 + Query（全量上下文）选材 + 双语润色
8. 生成草稿（选长度/语言）→ 局部重生成 → 多版对比
9. 好结果回填进 wiki（复利积累）
10. 选择模板 → 定稿
11. 导出 PDF（ATS版/多模板）/ DOCX / 求职信
12. 版本归档，供复用
```
