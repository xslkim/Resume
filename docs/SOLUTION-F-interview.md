# 方案 F：访谈驱动的素材挖掘

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`）并列 · 供横向比较
> **一句话**：知识库不靠表单录入，而靠 **LLM 主导的结构化访谈**来建与精炼——像招聘教练一样追问"你具体做了什么？有数字吗？结果如何？"，把回答挖成结构化、量化的成就素材。

---

## 1. 核心理念

A/B/D/E 都默认一个前提：**用户愿意、且会把素材录好**。现实里这个前提常不成立——素材又少又烂、几乎没量化（GR-3"量化优先"最难落地的根因）。

F 把差异放在**"获取方式"而非"存储方式"**：用对话主动挖掘，显著提升素材的**质量与完整度**。F 不与 A/B/D/E 互斥——访谈产出可落进 A 的 wiki 或 B 的原子。F 的价值在"进料口"。

---

## 2. 知识载体 / 数据模型

载体可复用 A 或 B（wiki / 原子库）。F 额外维护：
- **访谈状态**：已覆盖的经历、已追问的维度、待补的缺口清单。
- **缺口待办**：发现"这段经历缺数字/缺结果"时挂起，下次访谈主动补。

## 3. Ingest（录入/编译）= 访谈

```
1. 用户给出一段粗略经历（甚至一句话）
2. Interviewer-LLM 结构化追问：
     · 你具体负责什么？（厘清角色，防夸大）
     · 有没有数字/规模/指标？（逼出量化，服务 GR-3）
     · 结果/影响是什么？用了哪些技能？
3. 把回答抽取成结构化成就 + 双语化 → 写入 KB
4. 标记仍缺失的维度 → 进缺口待办，后续主动回访
```

> 关键护栏：访谈**只澄清不诱导造假**——追问是为了厘清真实细节，不得引导用户夸大；抽取出的内容仍须可溯源到用户原话（GR-1）。

## 4. Query（生成）

生成阶段沿用所选底层方案（A/B 等）的选材逻辑；F 的增益是"入库素材本身更优质完整"，从而生成质量更高、缺口更少。生成时也可继续对话式微调（"这段我想突出 X"）。

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **双语/长度/ATS**：取决于底层方案（A/B/E…）。
- **GR-3 量化**：访谈主动逼出数字，是所有方案里最能落实量化的。
- **FR-4.4 缺口**：在录入期就主动发现并回访补齐，而非等到生成才报缺。
- **GR-1 防夸大**：访谈 prompt 必须设"澄清而非诱导"的护栏。

## 6. 优势

- 素材质量天花板最高：主动挖掘 > 被动表单。
- 录入门槛低：说话比填表轻松，适合不擅自我表达者。
- 量化与缺口在**入口**就被解决。

## 7. 代价与风险

- **录入耗时**：多轮对话比填表慢，依赖用户配合度。
- **诱导造假风险**：追问设计不当会引导用户夸大，需强护栏。
- 访谈 prompt/状态机设计难，体验好坏高度依赖它。
- 不解决"知识怎么存/怎么选"——必须搭一个底层方案。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | F（访谈驱动） |
|---|---|---|
| 素材获取 | 用户被动手填 | LLM 主动访谈挖掘 |
| 量化/缺口 | 录入时靠提示，弱 | 入口主动逼出/回访 |
| 角色定位 | 完整方案 | 进料层（需搭底座） |
| 录入体验 | 填表 | 对话 |

## 9. 适合谁 / 何时选它

- 不擅长写简历/自我表达的用户。
- 高端定制场景，愿意花时间换高质量素材。
- 作为 A/B 的"前置进料增强模块"。

## 10. MVP 边界与工程量

- **MVP**：单条经历的引导式追问（角色/量化/结果三问）+ 抽取入库，底座先用 A 或 E。
- **后续**：跨经历缺口回访、画像导向访谈、生成期对话微调。
- 工程量：**中**（主要在访谈 prompt/状态管理；底座另算）。
