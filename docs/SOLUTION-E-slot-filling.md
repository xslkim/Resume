# 方案 E：确定性槽位填充（最少 AI）

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`）并列 · 供横向比较
> **一句话**：简历=一套**固定 schema 的槽位**；KB=带标签的内容池；选材靠**规则+标签匹配+打分**（可完全不依赖 LLM 选材），LLM 仅**可选地**用于措辞润色——最确定、最可控、最 ATS、成本最低。

---

## 1. 核心理念

A/B/C/D 都把 LLM 放在关键路径上。E 反其道：**把 AI 的参与度压到几乎为零**，用确定性规则跑通端到端。理由：求职者对"AI 编造"高度敏感（GR-1 是红线），而 ATS 解析、关键词覆盖、长度控制本质上都是**确定性问题**，不一定需要生成式模型。

E 也天然可作为其他方案的**保底降级层**（GR-6）。

---

## 2. 知识载体 / 数据模型

结构化内容池 + 固定简历模板槽位：

```
内容池（用户手填，结构化）：
  Bullet { text{zh,en}, tags[], metrics, section类型, impact }
  Skill  { name{zh,en}, tags[], 熟练度 }
  Experience / Education / Credential …（结构化字段）

简历模板（固定槽位 + 上限）：
  [Summary] [Skills(≤N)] [Experience(每段≤M条bullet)] [Education] [Credential]
```

---

## 3. Ingest（录入/编译）

几乎无"编译"：用户结构化录入即入池。双语可由用户填，或仅在此处调一次 LLM 翻译（唯一可选 AI 介入点）。无叙事生成 → 无漂移、无失真。

## 4. Query（生成）

```
1. JD → 关键词抽取（可用规则/词典，也可一次轻量 LLM）→ 关键词集
2. 标签/关键词匹配：内容池里命中 JD 关键词的 bullet/skill 打分
3. 按 section 规则填槽：每个槽位取分数最高的若干条，受条数与长度上限约束
4. 技能区按 JD 优先级重排（FR-5.6）
5.（可选）LLM 仅对填入的 bullet 做措辞润色贴合 JD，内容不变
6. 直接套 ATS 模板出 PDF
```

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **双语**：存双语字段，按 JD 语言取。
- **长度**：槽位+条数上限即长度控制，完全确定。
- **ATS**：模板固定为纯文本可解析；关键词覆盖由匹配规则直接保证（GR-4），绝不堆词造假。
- **GR-1**：内容全是用户原话，LLM 至多润色不新增事实 → 几乎零造假风险。
- **隐私**：可做到几乎不调线上模型（呼应可选的隐私收紧取向）。

## 6. 优势

- 最确定、最可复现、最可解释；用户几乎不需"信任 AI"。
- ATS 可控性、关键词覆盖最强。
- 成本最低、延迟最低、隐私最友好。
- 实现最简单，最快出 MVP；可作为其他方案的降级保底。

## 7. 代价与风险

- **灵活性最差**：贴合 JD 措辞能力弱（除非开 LLM 润色）。
- 机械感强、千篇一律，缺少"针对性综合"的惊艳感。
- 跨经历综合=无；详略只能靠规则粗调。
- 录入负担回到用户身上（结构化手填）。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | E（槽位填充） |
|---|---|---|
| AI 参与 | 核心（写+选+润色） | 极少（至多润色/翻译） |
| 选材 | LLM 黑箱 | 规则+标签打分（确定） |
| 真实性 | 尽力 + Lint | 近乎天然成立 |
| 灵活/惊艳度 | 高 | 低 |
| 成本/隐私 | 高/需上线模型 | 低/可几乎本地 |

## 9. 适合谁 / 何时选它

- 强烈不信任 AI 编造、要极致可控与 ATS 的用户。
- 隐私敏感、希望尽量少把数据交给线上模型。
- 作为其他方案的"降级保底层"。

## 10. MVP 边界与工程量

- **MVP**：结构化录入 + 关键词匹配 + 槽位填充 + 技能重排 + 单 ATS 模板。LLM 可完全不接。
- **后续**：可选润色层、词典扩展、多模板。
- 工程量：**低**（最快落地）。
