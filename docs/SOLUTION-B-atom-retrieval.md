# 方案 B：结构化成就原子库 + 检索-打分流水线

> 独立方案文档 · 与基线方案 A（`REQUIREMENTS.md`，LLM Wiki）刻意走相反路线 · 供横向比较
> **一句话**：不让 LLM 写知识，只让它抽取与最终润色；知识永远以确定性的结构化数据存在，靠"检索 → 打分 → 预算优选"选材，**每条简历 bullet 1:1 锚定一条原子**。

---

## 1. 核心理念

方案 A 把 LLM 当"wiki 作者 + 维护者"——LLM 写散文、可能漂移，需要 Lint 兜底。
方案 B 把 LLM 的生成范围压到最小：**只在两端用 LLM**（录入时抽取、出稿时改写选中的几条），中间的"知识存储 + 选材决策"全部是确定性、可审计的结构化流程。

知识不被"二次叙述"，所以**没有编译失真的空间**；选材是排序+优化，不是黑箱，所以**可解释、可复现**。

---

## 2. 知识载体 / 数据模型

不维护 wiki 页面，而是把一切归一化为**追加式、强类型**的记录。核心实体：**成就原子（Achievement Atom）**。

```
Atom {
  atom_id
  experience_id          # 所属经历（教育/工作/项目）
  raw_source_id          # 不可变溯源，指回原始录入（GR-1/GR-5 的结构性保证）
  canonical_text { zh, en }   # 双语平行字段（ingest 时 LLM 翻译）
  metrics [ {指标, 从, 到, 单位} ]  # 抽取出的量化成果（服务 GR-3）
  tags { 技能[], 行业[], 资深度, 软技能[] }
  embedding              # 语义向量
  impact_score           # 静态重要性（LLM 初评 + 用户可调）
  created_at / updated_at
}
```

经历实体（Experience）、技能（Skill）、证书/奖项同样是**结构化记录**，不是散文页。

---

## 3. Ingest（录入/编译）

1. 用户单语手动录入基础字段（同 A 的低门槛原则）。
2. LLM 做**抽取**：把自由文本拆成原子、抽出 metrics、打标签、翻译成双语平行字段。
3. 计算 embedding，写入向量索引。
4. **去重/合并**：新原子与既有原子做 embedding 聚类，疑似重复时提示合并（确定性 + 向量辅助，替代 A 的 Lint）。
5. 矛盾检测（时间冲突、机构信息不一致）走结构化校验规则（GR-2），命中即标记。

> LLM 在这里只做"抽取 + 翻译"，**不创作叙事**，所以不会引入素材之外的事实。

---

## 4. Query（生成）

确定性、可审计的流水线：

```
1. JD → LLM 抽取「需求向量」：技能 / 资深级别 / 行业 / 硬性要求 / 关键词
2. 检索：embedding 相似度 + 标签/元数据过滤 → 候选原子集（top-K）
3. 打分：混合排序 = 语义相似 + 关键词命中 + 时效 + impact_score，每条分数可展示
4. 选材：在「长度预算」（一页/两页）下做背包式优选，
        目标 = 最大化 JD 需求覆盖率（而非 LLM 凭感觉挑）
5. 组装 + 润色：LLM 只改写「被选中」的原子，使措辞贴合 JD 用词
6. 缺口分析「白送」：需求向量里未被任何原子覆盖的项 = gap（GR/FR-4.4）
```

---

## 5. 双语 / 长度 / ATS / 护栏 落地

- **双语**：存储即双语平行字段，出稿按 JD 语言直接取，无需现译。
- **长度**：长度=预算约束，直接进背包优选，确定性可控。
- **ATS**：选材已是结构化字段 → 排版模板纯文本可解析；关键词覆盖由"需求覆盖率"显式保证（GR-4），且只选真实原子不会堆词造假。
- **GR-1/GR-5**：每条输出 bullet 都带 `atom_id → raw_source_id` 链，溯源是**结构性成立**而非尽力。
- **GR-6 降级**：素材不足时输出覆盖率低但真实的版本，并在 gap 报告里说明。

---

## 6. 优势

- 真实性最强：知识从不被重新叙述，溯源是结构保证。
- 选材可解释、可复现：同 JD + 同库 → 同结果，能向用户解释"为何选/不选"。
- 无规模天花板：检索按需取，wiki 再大也不撞上下文窗口（解决 A 的开放问题 #6）。
- 缺口分析天然免费（覆盖矩阵的补集）。

## 7. 代价与风险

- **失去跨经历叙事综合**：A 能写主题摘要、连点成线；B 是机械拼装，跨经历洞察需工程化（主题标签或加可选综合层）。
- **工程量更大**：embedding / 向量库 / 排序器 / 预算优化器，起步成本高于 A 的"markdown + 一个 prompt"。
- 抽取质量决定上限：原子拆得好不好、标签准不准，直接影响检索。
- 复利形态是"数据越多检索越准"，少了 A 的"知识涌现"感。

## 8. 与方案 A 的关键差异

| 维度 | A（LLM Wiki） | B（原子库+检索） |
|---|---|---|
| 知识载体 | 叙事 wiki（散文） | 强类型结构化原子 |
| LLM 角色 | 作者+维护者（会漂移） | 抽取器+最终润色器 |
| 选材 | 全量塞上下文，黑箱挑 | 检索→打分→预算优选，确定性 |
| 规模上限 | 受上下文窗口约束 | 无窗口天花板 |
| 真实性 | 尽力溯源 + Lint 兜底 | 结构性溯源 |
| 维护 | Lint 修 wiki 漂移 | 去重/合并（半自动） |

## 9. 适合谁 / 何时选它

- 把"绝不造假 + 可解释 + 可复现"放在第一位。
- 预期素材量会很大、担心上下文塞不下。
- 团队有能力搭建检索/排序基础设施。

## 10. MVP 边界与工程量

- **MVP**：原子数据模型 + LLM 抽取 + 标签过滤 + 关键词/语义检索 + 简单加权打分 + 长度截断（背包可先用贪心）+ 单模板 PDF。
- **后续**：精细背包优化、impact 自动评估、聚类去重、覆盖矩阵可视化、回填复用。
- 工程量：**中高**（高于 A，主要在检索与排序栈）。
